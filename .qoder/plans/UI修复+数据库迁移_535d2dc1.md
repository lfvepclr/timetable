# 本地存储分层重构计划

## 目标

将当前 `local-db.js` 的单文件持久化（`timetable_db.json`）重构为分层存储架构，只改持久化层，查询引擎和业务逻辑（callFn）保持不变。

## 存储架构

| 数据 | 存储方式 | Key/路径 | 说明 |
|------|---------|----------|------|
| 配置数据 | wx.setStorageSync | `tt_config` | `{ courses: [], holidays: [], vacations: [] }` 合并存储 |
| 学生 | wx.setStorageSync | `tt_students` | JSON 数组，每条含 _id 作为关联键 |
| 课表 | wx.setStorageSync | `tt_lessons` | JSON 数组，students[].student_id 关联学生 |
| 课程包 | wx.setStorageSync | `tt_packages` | JSON 数组，student_id 关联学生 |
| 周模式 | wx.setStorageSync | `tt_patterns` | JSON 数组 |
| 用户 | wx.setStorageSync | `tt_user` | 单对象（老师信息、设置） |
| 课程反馈 | 文件系统 | `${USER_DATA_PATH}/feedbacks.json` | 内容较多，独立文件存储 |
| 上传文件 | 文件系统 | `${USER_DATA_PATH}/uploads/` | 照片、卡片图（已有实现） |

## 修改文件

仅修改 `miniprogram/utils/local-db.js` 的持久化层（约前 70 行 + 导出/导入函数），其余代码（查询引擎、CRUD 接口、callFn 业务逻辑）不变。

### 具体改动

**1. 替换常量定义（第 6-19 行）**

删除 `DB_FILE` 常量，新增存储映射：

```javascript
// 存储映射：table → { type, key, field? }
const STORAGE_MAP = {
  users:          { type: 'storage', key: 'tt_user' },
  students:       { type: 'storage', key: 'tt_students' },
  courses:        { type: 'storage', key: 'tt_config', field: 'courses' },
  packages:       { type: 'storage', key: 'tt_packages' },
  weekly_patterns:{ type: 'storage', key: 'tt_patterns' },
  lessons:        { type: 'storage', key: 'tt_lessons' },
  feedbacks:      { type: 'file',    path: `${wx.env.USER_DATA_PATH}/feedbacks.json` },
}
```

**2. 重写 init()（第 28-41 行）**

从多个源加载数据到内存 `db` 对象：
- `storage` 类型：`wx.getStorageSync(key)`，若有 `field` 则取 `config[field]`
- `file` 类型：`wx.getFileSystemManager().readFileSync(path)`
- 失败时初始化为空数组

**3. 重写 save() 和 flush()（第 43-67 行）**

引入脏表追踪（`dirtyTables` Set），只保存变更的表：
- `saveTable(table)`: 根据 STORAGE_MAP 写入对应目标
  - `storage` 类型：`wx.setStorageSync(key, data)`，若有 `field` 则先读 config 再更新字段
  - `file` 类型：`wx.getFileSystemManager().writeFileSync(path, JSON.stringify(data))`
- `save()`: 防抖 500ms，遍历 dirtyTables 逐个 saveTable，完成后清空
- `flush()`: 立即写入所有 dirtyTables

**4. 在 CRUD 操作中标记脏表**

`add()`、`update()`、`remove()` 调用 `markDirty(table)` 而非 `save()`：
```javascript
function markDirty(table) {
  dirtyTables.add(table)
  save()
}
```

callFn 内部的 `update()` 调用已会触发 markDirty，无需额外修改。

**5. 更新导出/导入（第 229-240 行）**

- `getRawData()`: 遍历所有表从内存 `db` 深拷贝（不变）
- `replaceData(newData)`: 遍历所有表，逐个 saveTable 写入对应存储目标，清空 dirtyTables

## 不变的部分

- `TABLES` 数组、`JSON_COLUMNS` 映射
- `matchValue()`、`matchWhere()` 查询引擎
- `normalizeRow()` 行规范化
- `query()`、`getById()`、`add()`、`update()`、`remove()`、`count()` 接口签名
- 全部 9 个 callFn 业务函数
- `api.js` 接口（无需修改）
- `app.js`（无需修改）
- 所有页面代码（无需修改）

## 关联关系设计

学生与课表的关联通过 `_id` 实现，无需额外索引：
- 学生记录：`{ _id: 'uuid', name: '张小明', grade: '三年级', ... }`
- 课表记录：`{ _id: 'uuid', students: [{ student_id: '学生uuid', status: 'scheduled' }], ... }`
- 课程包记录：`{ _id: 'uuid', student_id: '学生uuid', course_id: '课程uuid', ... }`

查询时通过内存 filter 匹配 `student_id`，性能足够（单老师数据量 <1000 条）。
